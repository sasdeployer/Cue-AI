# Phase 1 — Single-file hot-swap runtime

**Date:** 2026-07-21
**Status:** Approved direction; spec under review
**Scope:** Phase 1 ONLY. No video/Remotion, no AI image generation, no gallery
sandbox hardening. Those are later phases.

---

## Goal

Replace Cue's fragile remote-Sandpack preview with an **extremely fast, reliable,
same-origin runtime** that lets users **iterate** on a deck in real time. Three
pillars:

1. **Hot-swap runtime (client).** Pre-bake the fixed deck engine + component
   library into Cue's own bundle. Transpile only the generated `App.tsx` in the
   browser (Sucrase, ~1ms) and render it in a same-origin iframe. Theme changes
   swap a `<style>` with zero transpile.
2. **Iteration (client + server).** A follow-up prompt on a loaded deck **edits**
   it (not full regen). The result hot-swaps into the live preview, preserving the
   current slide.
3. **Reliability (server).** Before a deck reaches the client, the server
   compile-checks the single `App.tsx` against the fixed engine; on failure it
   feeds the error back to the model and retries (agentic self-correction). Kills
   the "blank deck from broken JSX" class.

### Why this shape
The engine + ~30 components are FIXED. Only `App.tsx` + `tokens.css` ever change.
So Cue is not a project bundler — it is a single-file hot-swapper. No Node in the
browser, no CDN, no WebContainer. See conversation for the full rationale.

## Architecture

```
Browser (Cue web app)                          Go server
┌───────────────────────────────┐             ┌──────────────────────────┐
│ Builder.tsx                    │  SSE        │ POST /api/decks (generate)│
│  ├─ chat (new deck | edit)  ───┼────────────▶│ POST /api/decks/:id/edit  │
│  └─ DeckPreview.tsx            │             │   ├─ llm.Generate/Edit    │
│       │ postMessage            │             │   ├─ CheckDeck (esbuild)  │
│       ▼                        │             │   └─ retry on error (≤2)  │
│  <iframe deck-runtime.html> ◀──┘             └──────────────────────────┘
│    ├─ registry (pre-baked engine + framer-motion + react)
│    ├─ Sucrase transpile(App.tsx) → require(registry)
│    ├─ mount <App/> into React root  (hot-swap on re-render)
│    └─ tokens.css → <style id=deck-tokens> (theme fast-path)
└───────────────────────────────┘
```

The engine is compiled ONCE by Vite (into the iframe bundle). Per deck/edit, only
`App.tsx` is transpiled and `tokens.css` is injected.

## Interface contracts (STABLE — parallel work builds against these)

### C1 — Module registry  `web/src/deck-runtime/registry.gen.ts` (generated)
```ts
// specifier -> module namespace object (always has .default; react also spreads named)
export const registry: Record<string, any>;
```
Keys include: `react`, `react-dom`, `react-dom/client`, `framer-motion`, and every
engine/component path as the model imports it: `./deck/Deck`, `./deck/Slide`,
`./deck/Build`, `./deck/Reveal`, `./components/Cover`, … (one per file in
`web/src/deck-template/src/deck` and `.../components`). Generated by
`scripts/gen-registry.mjs` (run in `dev.sh`, same pattern as the existing sync).
The require shim (C2) normalizes keys: strips leading `./`, strips `.tsx`/`.ts`,
matches with/without prefix. React/react-dom entries are `{ ...ns, default: ns }`
for CJS interop.

### C2 — Transpile  `web/src/deck-runtime/transpile.ts`
```ts
export function transpileDeck(appTsx: string):
  | { ok: true; App: React.ComponentType }
  | { ok: false; error: string };
```
Sucrase `transform(appTsx, { transforms: ['typescript','jsx','imports'], production: true })`
→ CJS. Execute with `new Function('require','module','exports', code)` where
`require` looks up the normalized specifier in `registry` (C1) and THROWS a clear
message on an unknown specifier (allowlist = security boundary for Phase 1).
Return `module.exports.default` as `App`. Any throw → `{ok:false,error}`.

### C3 — Runtime postMessage protocol (parent ⇄ iframe)
Parent → iframe:
- `{ type:'cue-render', appTsx:string, tokensCss:string }`
- `{ type:'cue-theme', tokensCss:string }`  // fast path, no transpile
Iframe → parent:
- `{ type:'cue-ready' }`                       // posted on load
- `{ type:'cue-ok' }`                          // rendered without error
- `{ type:'cue-error', phase:'compile'|'runtime', message:string }`
All messages are `postMessage(msg, '*')` between same-origin frames; validate
`type` prefix `cue-`. Types live in `web/src/deck-runtime/protocol.ts`.

### C4 — Server edit endpoint  `POST /api/decks/:id/edit`
Body `{ instruction:string, appTsx:string, tokensCss:string }`. Streams SSE with
the SAME event shape as generate (`delta` for chat prose, `error`, `done`). `done`
payload: `{ id, title, message, appTsx, tokensCss }`. Persists the updated deck
(overwrite `app_tsx`/`tokens_css`/`title` for that id). If `:id` unknown → 404.

### C5 — Compile check  `server/compile.go`
```go
// returns nil if the deck's App.tsx transpiles + all imports resolve against the
// fixed engine; else an error whose message is safe to feed back to the model.
func CheckDeck(appTsx string) error
```
Implemented with esbuild-Go (`github.com/evanw/esbuild/pkg/api`): build `App.tsx`
in-memory as entry, with `web/src/deck-template/src` (mirrored into
`server/reference/engine/` by dev.sh, or resolved via an on-disk path) as the
resolve root + `react`/`react-dom`/`framer-motion` marked external. Surfaces syntax
errors and unresolved imports (e.g. a hallucinated component). Prop-type errors are
out of scope for Phase 1 (esbuild doesn't typecheck).

### C6 — Client API  `web/src/lib/api.ts`
```ts
export function editDeck(id: string, instruction: string,
  current: { appTsx: string; tokensCss: string },
  h: GenerateHandlers): Promise<void>;   // same handler shape as generateDeck
```

## Data flow

**New deck:** prompt → `generateDeck` SSE → server: `llm.Generate` → `parseGeneration`
→ `normalizeDeckImports` → `CheckDeck`; on error retry (≤2) feeding the error →
`done` → Builder sets deck → DeckPreview posts `cue-render` to iframe → transpile +
mount → `cue-ok`.

**Edit (iteration):** deck loaded → follow-up prompt → `editDeck(id, instruction,
current)` → server edit prompt returns full updated `App.tsx`+`tokens.css` →
`CheckDeck` + retry → `done` → Builder updates deck → DeckPreview diffs props: if
only `tokensCss` changed → post `cue-theme` (instant, slide preserved); else post
`cue-render` (transpile + re-render; iframe kept mounted so `location.hash` restores
the current slide via the engine's existing hash sync).

**Error:** `cue-error` → DeckPreview shows an inline overlay with the message (never
a silent blank). Server errors already surface via SSE `error`.

## Risks & mitigations
- **React 19 + framer-motion:** engine was written for React 18. Pin a
  framer-motion version that supports React 19; verify no runtime warnings. (Verify step.)
- **Keyboard/focus isolation:** the engine binds global `keydown` (arrows advance
  slides). The iframe isolates this from Cue's chat. Confirm typing in chat never
  advances slides.
- **iframe sizing:** engine uses `100vh`/`100svh`; iframe must be `width/height:100%`
  of the preview pane. Confirm no clipping.
- **Security (Phase 1 boundary):** require-shim allowlist means generated code can
  only import the fixed engine — no arbitrary modules. Full cross-origin isolation
  for the public gallery is deferred (documented follow-up).
- **esbuild resolve root:** dev.sh must place the engine where `compile.go` resolves
  it; keep it in sync with the rsync target.

## Acceptance criteria (local tests — run via `./dev.sh`)
1. `web` builds (`npm run build`) and `server` builds (`go build ./...`) with no errors.
2. Landing prompt → deck renders in the preview iframe within ~1s of `done`
   (screenshot shows real slides, not blank; same-origin so capture works).
3. Existing gallery decks (ARCSEC, RapidTransit) load and render via the new runtime.
4. **Iteration:** with a deck loaded, "make the theme blue" updates colors
   **instantly** (theme fast-path, no full rebuild) and keeps the current slide.
5. **Iteration:** "add a closing slide" re-renders with the new slide present.
6. **Reliability:** a deliberately broken `App.tsx` (bad import) is caught by
   `CheckDeck` and retried, OR surfaces a clear error overlay — never a blank pane.
7. No `codesandbox.io` / jsdelivr requests remain for preview (verify Network).
8. Typing in chat does not advance the deck.

## Out of scope (Phase 1)
